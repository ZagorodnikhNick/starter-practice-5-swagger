## Практика 3. JSON + API + Postman

### Цель (что нужно уметь по итогам)

1. Понимать, что такое **JSON** и где он используется в API.
2. Уметь отправлять HTTP‑запросы через **Postman** (или альтернативно через `api.http`).
3. Протестировать:

- **свой API** (`/api/products`) — минимум 3 запроса,
- **внешний API** — минимум 5 запросов.

Требование по внешнему API и количеству запросов — см. методичку преподавателя.

---

### Часть A — запускаем backend

1. Откройте папку `backend/`.
2. Установите зависимости:

```bash
npm i
```

3. Запустите сервер:

```bash
npm run dev
```

4. Проверьте, что сервер отвечает:

- В браузере: `http://localhost:3000/`
- Или запросом в Postman: `GET http://localhost:3000/`

---

### Часть B — Postman: быстрый чек-лист

> Если Postman не получается — используйте файл `backend/api.http` через VS Code + расширение **REST Client** (по согласованию с преподавателем).

1. Откройте Postman → **New** → **HTTP Request**
2. Создайте запрос `GET http://localhost:3000/api/products` → нажмите **Send**
3. Создайте запрос `POST http://localhost:3000/api/products`
   - вкладка **Body** → **raw** → **JSON**
   - вставьте пример JSON (можно свой):

```json
{
  "title": "Печенье",
  "category": "Сладости",
  "description": "Тестовый товар",
  "price": 79,
  "stock": 10
}
```

4. Создайте запрос `PATCH http://localhost:3000/api/products/<id>` (id возьмите из ответа GET)
5. Создайте запрос `DELETE http://localhost:3000/api/products/<id>`

---

### Часть C — внешнее API (минимум 5 запросов)

Сделайте **5 запросов** к внешнему API и сохраните результаты (скриншоты).

Подсказка: пример внешнего API есть в `backend/api.http` (JSONPlaceholder, OpenWeatherMap и т.п.).  
**Важно:** если внешнее API требует ключ — получите ключ и храните его локально (в `.env`, не коммитьте в git).

---

### Что сдаём по практике 3

- Ссылка на репозиторий.
- Скриншоты выполненных запросов (можно в папку `docs/screens/`).
- Минимум:
  - 3 запроса к своему API,
  - 5 запросов к внешнему API.

## Практика 5. Swagger (OpenAPI) — документация REST API

### Цель

Подключить Swagger к вашему Express‑серверу и **задокументировать API** так, чтобы документация была доступна в браузере и позволяла отправлять тестовые запросы (**Try it out**).

Важно: в учебном репозитории пример документации **неполный** — к сдаче вы должны:

1. дописать недостающие запросы (если у вас их нет в коде), как в прошлых практиках;
2. **задокументировать все CRUD‑операции** в Swagger.

### Установка зависимостей (backend)

Перейдите в папку `backend/` и установите пакеты:

```bash
npm i -D swagger-jsdoc swagger-ui-express
```

### Подключение Swagger в `backend/app.js`

1. Подключите зависимости:

- `swagger-jsdoc` — генерирует OpenAPI‑спеку из JSDoc‑комментариев
- `swagger-ui-express` — показывает интерактивный UI в браузере

2. Создайте `swaggerOptions`, сгенерируйте `swaggerSpec` и подключите Swagger UI по адресу **/api-docs**.

После запуска сервера документация должна открываться по адресу:

- `http://localhost:3000/api-docs`

### Где писать Swagger-аннотации

Аннотации пишутся в виде JSDoc‑комментариев `@swagger` рядом с роутами.
В этом проекте пример находится в файле:

- `backend/routes/products.js`

### Схема

    1.	OpenAPI — формат/стандарт “паспорта API”.
    2.	Вы пишете паспорт в коде через @swagger (или можно отдельным YAML/JSON файлом).
    3.	swagger-jsdoc превращает комментарии → OpenAPI JSON.
    4.	swagger-ui-express показывает этот JSON как веб-документацию (/api-docs).

Если у вас вопрос “как правильно описать эндпоинт/параметры/тело запроса/ответы/коды ошибок/схемы данных” — ответ в спецификации.

swagger-jsdoc: https://github.com/Surnet/swagger-jsdoc
Для чего: это библиотека, которая генерирует OpenAPI JSON из ваших JSDoc-комментариев @swagger в коде.
То есть именно она читает такие блоки в routes/products.js:

	•	разобраться, почему Swagger UI “не видит” ваши эндпоинты (часто проблема в apis путях)
